Design Is A Process

Orbiting since: Jan 10, 2014 by Stratos  | Tags: design, communication, Responsive  | Comments (0)

If you don't know what you're talking about, communicate that. If you're uncomfortable with any aspect, communicate that. And keep doing that along this process because there is always going to be unknowns and unforeseen scenarios popping up that need to be addressed. Address them with communication.

Design is a process, and the process involves you.

I cannot recall ever getting a design "right" the first time.

I start by cleaning up your existing content, or creating placeholders for your content. Doing this keeps the HTML code clean.

The layout uses a framework that controls the way your website looks. It will look good on all devices from smartphones, tablets, iPads, laptops, and regular desktops.

That's one website layout built to handle anything thrown at it. It's called Responsive because it responds to the width of the screen.

I start out with boring because usually I'm given fixed width layouts.

Fixed widths mean that the designer can do whatever the hell they want; the website will look pretty good on a laptop or desktop browser.

But show me the same website on a smartphone and you just might feel like you're looking at eye exam or need one after looking at it.

I am often guilty of explaining the obvious. Because each of us tend to see what we want to see, I am forced to be objective; I get to play devil's advocate.

Case #1

Let's start off with a recent project, a failure all the way around because I didn't communicate that design is a process. Or I did and the client didn't get it. Hence this post. I'm not pointing the finger at anyone but my self. And hopefully the owner has more sense than to badmouth me.

How it looks on my tiny iPhone:


Were you able to tell what business this website is in? If you guessed flies, you were close. Insects? Getting colder. Look at the Nav menu on the left. Can you guess it now?

How a "boring" Responsive design might look:


Now you know right away what the website is about; I don't have to explain it. You got it right away.

I could've done a better job, yes, but I did what the client wanted. Later I found out that the client really didn't want that. And the design aspect of this is easy enough: make the text smaller on the phone. But that doesn't really matter at this point.

That's because I was in the middle of adding in graphics so it resembled the fixed width site. There's no room on the iPhone screen width for the honey comb graphics. On the tablet, yes, and on desktop, of course. It was very easy to drop these from the iPhone with just a single class declaration:

class="hidden-xs"

What that tells the browser is do not display on extra small devices. And the controlling code goes into a simple width controller called "media query". The browsers are very aware of how wide and tall your screen is at any given nanosecond. Change the width and it responds to that. It uses the style sheet to control it. Beautiful!

Lesson learned: it's always a fine line that I walk between giving what the client wants versus what they need.

Lesson for all would-be clients: design is a process. Communication is a two-way street. If you don't know what you're talking about, communicate that. If you're uncomfortable with any aspect, communicate that. And keep doing that along this process because there is always going to be unknowns and unforeseen scenarios popping up that need to be addressed. Address them with communication.

Website technology continues to evolve. I'm not on the leading edge of it by any means, but I do know the importance of effective marketing. That hasn't changed.

What has are the methods of marketing. Fixed width websites will never retain eyeballs the way Responsive does.

If you care enough about your business's long term future, you'll make your website Responsive.
